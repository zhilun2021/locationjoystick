# Wiki Contributing Guide

## General writing guidelines

All wiki pages are read by **app users, not developers**. Write every sentence as if the reader has never seen a line of code.

**Rules that apply to every page:**

- No code symbols, class names, method names, permission constants, or file paths — ever. If you need to refer to something technical, describe what it does instead.
  - Bad: `MockLocationService` keeps running in the background.
  - Good: The app keeps running in the background.
  - Bad: The `hot_` ID prefix identifies entries added by this feature.
  - Good: The entries added by this feature are removed when the toggle is turned off. Your own entries are never deleted.
- No internal Android or programming concepts: no "service", "foreground service", "StateFlow", "ViewModel", "dependency injection", "SQL database", "HTTP", "DHCP", "annotation processing", "serialization", "reactive streams", etc. Replace with plain English.
  - Bad: A foreground service runs location updates at 1 Hz.
  - Good: Location updates run continuously in the background.
- No library or framework names in descriptions unless the library itself is the subject (e.g. the Acknowledgements page). Even then, describe what it does for the user, not how it works internally.
- No XML tag literals, code paths, or module paths.
- `<code>` is allowed **only** for UI labels the user must type or read verbatim (e.g. `geo:` links, URL parameters like `lat` and `lon`).
- Active voice, present tense. One idea per sentence.
- Audience test: could a non-technical friend understand every sentence? If not, rewrite.

## Adding a new page to the navbar

All nav items live in `docs/wiki/wiki-init.js`, in the `NAV_ITEMS` array near the top of the file.

```js
var NAV_ITEMS = [
  { href: 'index.html',       label: 'Quick Start' },
  { href: 'home.html',        label: 'Home' },
  // ...
  { href: 'changelog.html',   label: 'Changelog' },
  { href: 'privacy.html',     label: 'Privacy' },
  { href: 'acknowledgements.html', label: 'Acknowledgements' },
];
```

Steps:

1. Create `docs/wiki/<page>.html` following the same shell as any existing page (DOCTYPE, head with `style.css` + `@docsearch/css@4`, `<div class="layout"><main>…</main></div>`, `<script src="wiki-init.js"></script>`).
2. Add `{ href: '<page>.html', label: '<Label>' }` to `NAV_ITEMS` at the position where it should appear in the sidebar. Ordering convention: feature pages first (alphabetical within a group), then meta pages (Changelog, Privacy, Acknowledgements) at the end.
3. No other file needs to be edited — `wiki-init.js` injects the sidebar into every page at runtime.

External links (GitHub, issue tracker) go in `EXT_ITEMS` below `NAV_ITEMS`, not in `NAV_ITEMS`.

## Maintaining `changelog.html`

### Source of truth

`CHANGELOG.md` at the repo root is the exhaustive changelog generated by conventional commits tooling. `changelog.html` is a curated, user-facing distillation of it — not a mirror.

### When to update

Update `changelog.html` whenever a new version is added to `CHANGELOG.md` (i.e. a new `## [x.y.z]` section appears).

### Structure

Each version is an `<h3>` whose text is a link to the GitHub release, followed by a `<ul>`:

```html
<h3><a href="https://github.com/shortcuts/locationjoystick/releases/tag/vX.Y.Z"
       target="_blank" rel="noopener">X.Y.Z</a> — Month DD, YYYY</h3>
<ul>
  <li>…</li>
</ul>
```

- Versions appear in descending order (newest first).
- The release URL pattern is `https://github.com/shortcuts/locationjoystick/releases/tag/vX.Y.Z`.
- Date format: `Month DD, YYYY` (e.g. `June 13, 2026`). Derive from the CHANGELOG.md heading date `(YYYY-MM-DD)`.

### Content rules

**Maximum 5 bullets per version.** Prioritise in this order:
1. New user-visible features
2. Significant UX improvements
3. Fixes for problems users would notice (crashes, wrong icons, broken import)

**Skip entirely:**
- Pure maintenance releases (version bump only, trigger-release commits, lint fixes) — merge them silently into the previous version's section, or omit the version heading if there is nothing user-visible.
- Internal refactors, test changes, CI/build changes, code style fixes.
- Fixes so minor they would not confuse a non-technical user (e.g. "resolve all Kotlin compiler warnings").

**Writing style:**
- Audience is non-technical app users — no code terms, module names, class names, or Android internals.
- One sentence per bullet, active voice, present tense describing what the app now does.
- Bad: "EphemeralReplayController.addWaypoint() now passes followRoads flag through the chain."
- Good: "'Add next point' while walking now follows roads if the original walk used roads."
- Use plain `<code>` only for UI label names (e.g. `<code>geo:</code>` URIs the user types), never for code symbols.

### Example

```html
<h3><a href="https://github.com/shortcuts/locationjoystick/releases/tag/v1.0.0"
       target="_blank" rel="noopener">1.0.0</a> — January 1, 2027</h3>
<ul>
  <li>Routes can now be exported as GPX files from the route detail screen.</li>
  <li>The floating widget remembers its last position on screen between sessions.</li>
  <li>Fixed: tapping a favorite while roaming no longer crashes the app.</li>
</ul>
```
